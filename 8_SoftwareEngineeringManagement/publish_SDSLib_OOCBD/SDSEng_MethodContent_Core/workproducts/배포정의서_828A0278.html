Main Description

작성항목

  1. 업무명 : 배포파일이 담당하는 업무명을 입력한다. 프로젝트내의 공통업무와 관련된 부분에는 “공통”이라는 접두어를 사용하여 지정한다.
    • 예) 업무공통, 시스템공통
    • 배포파일의 유형이 WAR인 경우 업무명은 ‘웹’을 접미어 사용한다.
  2. 배포파일명 : 클래스를 묶는 물리적인 배포 파일명을 기술한다. 이때 배포파일의 유형은 JAR, WAR, EAR 등이 될수 있다.
    • 예) JAR : 일반적인 Java 패키징 단위
           EJB JAR : EJB       어플리케이션 패키징 단위
           WAR :       웹 어플리케이션 패키징 단위. 굳히 배포파일로 묶지 않고 배포를
                           하는경우가 많다.
           EAR : J2EE       어플리케이션 패키징 단위, 관련 라이브러리 JAR를 모두 포함한다.
  4. 그룹명 : 배포파일내에 구분되는 그룹명을 입력한다. 컴포넌트를 사용하는 경우에는 컴포넌트명을 사용하며, 그렇지 않은 경우에는 기능별 그룹단위를 지정하여 사용한다. 기능별 그룹단위는 분석패키지를 기준으로 정의한다. 배포파일내에 관련클래스가 많은 경우 관리상의 편리성을 위하여 이 그룹단위로 jar 파일을 작성하여 패포파일명으로 묶는것도 가능하다. 배포파일명이 ‘웹’인 경우 패키지명이 될수있다.
  5. 관련소스 : 관련 소스를 기술한다. 해당 그룹에 있는 모든 파일을 포함한다면 이를 일일이 기술할 필요는 없이 관련 패키지를 기술한다. 웹의 경우 필요에 따라 *(모두포함)나 해당 JSP를 적어준다.
  6. 관련설정파일명 : 배포관련 정보를 가지는 설정파일명을 기재한다.

Key Considerations

작성시 고려사항

  • 배포정의서를 문서로 작성한다면 표준문서 양식에 따라 작성한다. 그러나 모델상에 Deployment Diagram으로 표현한다면 각 항목이 다이어그램에 표현되었는지를 확인한다.
  • 배포 파일에 대한 명명기준을 정의해야 한다.
  • 배포파일에 포함되는 소스파일은 작성과 변경이 용이하도록 파일 각각을 명시하기 보다는 패키지만 명시하도록 한다.
  • 프레임워크를 기반으로 개발하는 경우 관련 설정파일은 필수적이다. 아키텍처 정의서와 개발표준을 기준으로 작성하도록 한다.
  • 트랜젝션 관리를 어떻게 하느냐에 따라서 원격호출, 로컬호출을 고려하여야 한다. 아키텍처 정의서를 참고한다.
  • 물리적인 환경(기술아키텍처)을 고려하여 몇대의 서버에, 몇 개의 인스턴스를 사용할지 고려햐야 한다. 아키텍처 정의서를 참고한다.
  • 배포파일이 너무 많게 되면 이를 관리하는 것이 어렵다. 갱신주기가 서로 다른 모듈은 분리하여 패키징한다.
  • 웹 어플리케이션의 구성요소는 서버컴포넌트와 많이 다르기 때문에 이런 차이를 고려해야 한다. 웹의 경우 관리를 위하여 패키징 하지 않고 배포하기도 한다.
  • 배포파일간의 상호참조가 존재하는 경우 새로운 모듈을 추가하여 이를 제거한다.
  • 성능을 고려하여 연관성이 강한 업무는 하나의 배포단위로 패키징한다.
  • 단순하게 모든 컴포넌트를 하나로 패키징 할수 있다. 패키징이 단순하며 관리가 용이하다는 장점이 있지만, 재사용성이 떨어진다.
  • 각각의 컴포넌트를 개별적으로 패키징한다. 개별적으로 업그레이드가 가능하기 때문에 재사용성이 좋아지지만 배포방법이 복잡해지고 모듈간에 버전관리 문제가 발생할수 있다.
