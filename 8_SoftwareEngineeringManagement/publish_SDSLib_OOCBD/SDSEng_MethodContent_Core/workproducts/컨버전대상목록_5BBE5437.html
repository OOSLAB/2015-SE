Artifact: 컨버전대상목록
기존 시스템의 데이터를 기준으로 컨버전 대상이 되는 데이터의 이름, 데이터 속성 등을 식별하고 컨버전의 우선순위를 결정한다. 컨버전 비대상이 되는 데이터는 고객과 협의하여 사유를 명시하여 컨버전 과업의 범위를 확정한다.

Description
Main Description

작성항목

  1. 시스템명 : 기존 시스템의 이름. 대규모의 시스템일 경우에는 서브시스템명을 사용한다.
  2. 테이블(데이터)ID : DB인 경우에는 테이블명, 파일인 경우에는 파일명을 적는다.
  3. 테이블(데이터)명 : 해당 데이터에 대한 설명
  4. 건수 : 기존 시스템의 데이터중 전환대상인 건수를 표시한다. 건수는 개략적인 컨버전 볼륨을 산정하기 위한 기초자료로 사용되므로 가능하다면 각 테이블 별로 사이즈를 산정해서 자료항목을 추가하는 것이 더 좋다.
  5. 데이터 구분 : 테이블(데이터)의 성격을 명시한다. 예) 코드, 마스터, 일반, 로그, 통계, Binary파일, 이미지파일, 임시테이블 등으로 구분한다.
    • 코드 – 코드 데이터. 속성상 데이터에 대한 입력,수정,삭제가 거의 없으므로 사전에 컨버전을 수행할 수 있다.
    • 마스터 – 업무의 중심이 되는 테이블. 데이터 모델상으로는 각 주제영역의 핵심 엔티티가 마스터이며, ER다이어그램 상에서는 일반적으로 다른 테이블과 1:N의 관계를 가진다.
    • 일반 – 마스터와 1:N의 관계를 가지는 테이블로 단순한 로그성 데이터가 아닌 의미를 갖는 데이터. 일반적으로 우선 순위는 마스터 테이블과 로그 테이블의 중간에 위치한다.
    • 로그 – 트랜잭션 로그 테이블. 업무에 직접적으로 사용되는 데이터를 제외하고는 대부분 사후 컨버전이 가능하다.
    • 통계 – 통계처리를 위한 집계 테이블. 사전 컨버전이 가능하다.
    • 파일 – 이미지 파일, 첨부 파일 등의 데이터로 데이터베이스에 저장되지 않은 형태의 데이터.
    • 아카이브 – 신 시스템에서 업무적으로 사용하지 않는 데이터이나 유사시 참조를 위해 스키마의 변동없이 단순 마이그레이션하는 데이터.
    • 임시데이터 – 데이터베이스 작업, 백업 등을 위해 업무와 무관하게 임시로 생성한 데이터. 일반적으로 전환대상에서 제외한다.
  6. 전환여부 : 전환대상인 경우에는 Y, 비대상인 경우에는 N으로 표기한다.
  7. 우선순위 : 컨버전의 우선순위를 표시한다. 데이터의 의존적, 시간적 우선순위를 의미하며 일반적으로 코드, 데이터, 통계/로그 순으로 정하며 데이터는 의존관계를 고려하여 마스터, 일반 테이블 순으로 정한다. 오류데이터 정비의 우선순위와 컨버전 우선순위가 다른 경우에는 ‘정비 우선순위’를 별도의 항목으로 추가하여 작성할 수 있다.
  8. 비고 : 컨버전 비대상인 경우에는 사유를 명시하며, 컨버전 대상 데이터에 대한 특이사항이 있는 경우에 명시한다. 예를 들면, ‘2000년 1월 1일 이전 자료는 전환하지 않음’.

Key Considerations

작성시 고려사항

  • 데이터 건수는 해당 테이블의 전체건수보다는 컨버전 대상이 되는 건수를 표시한다. 예를 들어 20년간의 데이터가 있다면, 최근 5년간의 데이터만을 컨버전하고 이전 데이터는 버리거나, 본 컨버전에서 최근 3년간의 데이터만 컨버전하고 사후 컨버전에서 이전의 20년간의 데이터를 컨버전하는 경우이다.
  • 우선순위는 컨버전 우선순위를 의미하며 정비대상인 경우에는 일반적으로 정비우선순위와 동일하다. 만일 컨버전 우선순위와 오류데이터 정비 우선순위가 다른 경우에는 산출물 양식에‘정비 우선순위’ 항목을 추가하여 작성한다.
  • 데이터가 변하지 않는 Read-Only성 데이터는 사전 컨버전 데이터로 분류하고, 업무에 직접적으로 필요하지 않는 데이터는 사후 컨버전 데이터로 분류한다. 이 값은 향후 컨버전 계획서에서 활용한다.
  • 사전, 사후, 본 컨버전에 수행할 데이터를 분류한 내용은 후에 컨버전계획서를 작성할 때 시간적인 제약을 고려하여 확정된다. 본 단계에서는 시간적인 제약을 고려하여 사전, 사후 컨버전 대상을 정의하기 보다는 데이터의 성격에 따라서 정의한다.
Tailoring
Impact of not having

미작성시 영향

  • 기존 시스템의 데이터를 기준으로 컨버전 대상을 정의한 문서가 없기 때문에 기존 시스템 데이터의 컨버전 누락여부를 확인하기 어렵다.
Reasons for not needing

작성이 불필요한 경우

  • 컨버전 대상이 없는 경우 생략할 수 있다.
  • 스키마의 변동없이 데이터만 마이그레이션하는 경우에도 임시 데이터 등을 제거해야 하기 때문에 필요하다.
Representation Options

제출시 고려사항

  • 컨버전 대상, 정비 대상 등에 대한 고객의 승인을 득해야 한다.
