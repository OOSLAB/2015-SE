Work Product (Artifact): 통합테스트시나리오
통합 테스트 시나리오는 개발된 시스템의 데이터 및 업무기능이 정상적으로 수행되는지를 테스트하기 위한 시나리오이다. 요구사항정의 중 기능적 요구사항을 정의하는 유스케이스 목록 및 유스케이스모델(다이어그램/정의서), 사용자 인터페이스 요구사항, 기타 요구사항을 바탕으로 통합테스트시나리오를 작성한다.

Main Description

주요 작성항목

  1. 작성자/작성일 : 작성내용이 명확하지 않은 경우 작성자에게 작성내용의 의미를 확인하고, 작성내용에 대한 작성일을 확인하기 위한 목적으로 작성자와 작성일을 기재한다.
  2. 대표 시나리오 ID : 관련 유스케이스들을 커버할 수 있는 대표 시나리오 ID를 입력한다.
  3. 시스템명 : 해당하는 서브시스템의 이름을 입력한다. 여러가지 서브시스템에 걸쳐 작성된 경우 대표 업무명을 입력 한다.
  4. 작성자/작성일 : 시나리오를 작성한 작성자와 최종 작성 완료된 작성일을 기록한다.
  5. 관련 Usecase ID : 본 시나리오에 포함되는 유스케이스들의 ID를 입력한다.
  6. 테스트 수행자/수행일 : 테스트수행자의 성명을 입력하고 결함수정, 재테스트를 포함하는 테스트 완료일을 입력한다.
  7. 테스트 차수 :여러 차수에 걸쳐서 테스트를 수행할 때에 필요하며, 1차인 경우 1, 2차인 경우 2의 형태로 기록하도록 한다.
  8. 관련 Usecase 명 : 본 시나리오에 포함되는 유스케이스들의 명을 위에서 입력 Usecase ID 순서대로 입력한다.
  9. 테스트사전조건 : 해당 시나리오 ID의 테스트가 수행되기 위한 전제 조건을 기록한다. 없는 경우 공란으로 두어도 무방하다.
  10. 시나리오 ID : 관련 유스케이스 기준으로 한 건 씩 시나리오 ID를 입력한다.
  11. # : 시나리오 ID에 해당하는 케이스 번호를 입력한다. 시나리오 ID가 바뀌면 1번 부터 다시 일련번호를 입력한다.
  12. 화면명 : 테스트 대상이 되는 화면명을 입력한다.
  13. 테스트절차 : 해당 테스트케이스에 대한 테스트절차를 기술한다.
  14. 테스트 데이터 : 테스트 조건(입력값)을 입력한다. 테스트케이스 절차는 동일하나 테스트데이터가 다수의 케이스를 갖는 경우 별도의 시트를 작성하여 연결한다.
  15. 예상결과 : 예상결과를 입력한다. 시험 데이터를 별도로 작성한 경우 해당 시트에 예상결과도 함께 표시한다.
  16. 테스트 결과 : 테스트 수행 결과 정상인 경우에는 "P", 결함이 발생했을 경우에는 결함번호를 기록할 공간으로, 시나리오 작성 시에는 공란으로 둔다.
  17. 비고 : 테스트 수행하기 위해 참고할 만한 사항을 기록한다.

Key Considerations

작성시 고려사항

  • 여러 개의 유스케이스들의 조합으로 구성되는 비즈니스 흐름에 대한 테스트시나리오는 유스케이스 크기 수준으로 정의된 액티비티의 흐름을 나타내는 액티비티다이어그램에서 정의된 비즈니스 프로세스 수준으로 작성하며, 유스케이스 ID와 유스케이스명에 포함된 유스케이스 ID와 유스케이스명을 기술한다.
Tailoring
Impact of not having

미작성시 영향

  • 시스템이 전체 기능적 요구사항을 포함하고, 요구사항에 따라 정상적으로 구현 되었는지 검증하기 어렵다.
  • 단위업무간 처리흐름을 거친 데이터의 정확성을 검증하기 어렵다.
  • 테스트를 통해 발견된 에러를 재현하기가 어렵다. 따라서 초기 결함이 발견된 순서대로 테스트를 재 수행할 수 없으므로 결함이 수정된 후에도 이를 확신할 수 없다.
Reasons for not needing

작성이 불필요한 경우

  • 통합테스트 시나리오는 반드시 작성하도록 한다.
Representation Options

제출시 고려사항

  • 설계단계 완료 시 고객의 승인을 득하여 테스트계획서의 베이스라인을 정한 후 변경관리 한다.
