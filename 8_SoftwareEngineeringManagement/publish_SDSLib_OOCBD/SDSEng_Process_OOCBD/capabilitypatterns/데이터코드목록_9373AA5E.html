Work Product (Artifact): 데이터코드목록
효율적인 데이터 처리를 위해 코드화하여 관리할 필요가 있는 항목들을 도출하고, 도출한 전체 코드들에 대한 목록을 문서화한다.

Main Description

작성항목

  1. 시스템명
  2. No. : 코드의 순서를 일련번호로 기재
  3. 코드구분 : 물리적인 데이터 코드 관리 유형에 대한 분류를 기재. 독립된 물리적 테이블로 관리할 코드와 하나의 통합된 테이블로 관리할 코드 유형을 기재
    1. 단독코드 – ‘단독’으로 기재
      • 독립적인 별도의 테이블을 생성하여 관리하는 코드
      • 코드/코드 값 이외의 추가적 관리 속성이 존재
      • ex) 지역코드, 조직코드, 예산과목코드 등
    2. 통합코드 – ‘통합’으로 기재
      • 일반적으로 코드/코드값만 관리하는 수십개 이내의 코드
      • ex) 거래처구분, 사용자유형, 사용자상태 등
  4. 코드분류명 : 코드 분류 명칭
  5. 코드분류ID
    1. 단독 코드인 경우
      • 해당 코드가 주 PK인 테이블ID를 기재
    2. 통합 코드인 경우
      • 코드분류를 기재
  7. 체계 : 데이터 코드를 표기하는 문자(또는 숫자)의 구성체계
  8. 길이 : 관리하고자 하는 코드의 길이를 바이트수로 기재.
  9. 설명 : 데이터 코드의 용도 및 발생시점 등의 특성을 간략히 기술.

Key Considerations

작성시 고려사항

  • 동일한 코드에 대해 각 업무별로 별도의 코드로 중복 정의할 경우, 코드에 대한 일관성 유지가 어려울 수 있으므로 각 업무담당자들은 협의를 통해 코드를 정의하여 중복 정의되지 않도록 한다.
  • 코드의 길이를 현재 코드수만을 기준으로 너무 타이트하게 설계하면, 향후 코드 확장 시 코드 설계를 수정해야 하는 등의 문제가 있을 수 있으므로 향후 확장성을 고려하여 코드의 길이를 정의하도록 한다.
  • 작성항목 중 체계를 입력 시 해당 코드의 설계내용을 직관적으로 이해할 수 있도록 한다. 체계 입력에 대한 예제는 다음과 같다.
    • 복합의미가 포함된 코드인 경우
      • 여러 컬럼으로 구분된 경우 ","으로 구분한다.
      • 한 컬럼 내 그룹이 나뉘는 경우 "+"로 구분한다.
    • 그룹화된 특정의미 뒤에 C# 또는 N#을 괄호안에 표시함
      • C는 알파벳으로 이루어 진 경우이고, N은 숫자로 이루어진 경우이며 뒤의 #은 자리수를 의미함
      • 예1) 거래유형코드 -> 업무구분코드(C2), 거래유형코드(N2) (거래유형코드는 업무구분코드와 거래유형코드 두개의 컬럼으로 구성되며, 업무구분코드는 영문으로 이루어진 2자리 코드이고, 거래유형코드는 숫자로 이루어진 2자리 코드임)
    • 일반적으로 단순하게 01~99처럼 부여하는 코드인 경우
      • 일련번호(N2) 형태로 기입한다.
      • 임의번호, 임의의값, 임의숫자, 알파벳 같이 표현하지 않음
    • A~Z 또는 약어 형태의 영문으로 이루어진 코드의 경우 영문자(C#) 형태로 기술한다.
      • 예) 업무구분코드가 AC,RM,CM 같은 경우 영문자(C2)로 기술
    • 스페이스를 사용하는 경우 SPACE(3) 처럼 표현하고, 특정문자를 사용하는 경우 "X"처럼 쌍따옴표로 구분하여 사용한다.
      • 예) 어떤 코드가 A01, A02, A03처럼 구성되는 경우 : "A"+일련번호(N2)
    • 그룹화된 특정의미를 분명하게 기술하도록 한다.
      • 예) 1레벨 -> ~1레벨, 구분코드 -> ~구분코드
    • 코드 데이터들이 Row 그룹별로 특정 의미를 갖는 경우 구성체계 항목에 기입하되, 같은 칸 아래 줄에 표현
      • 예) 재무제표주석구분코드가 일련번호(N4) 인 경우
        • 0100 ~ 0199 : 사용제한 관련 코드
        • 0200 ~ 0299 : 유가증권 관련 코드
        • 0300 ~ 0399 : 장기투자증권 관련 코드
  • 코드의 성격을 정확히 파악하여 단독코드 또는 통합코드로 분류하여 설계하도록 한다.
  • 데이터 코드의 코드체계에 정의한 전체 길이와 코드 길이가 일치하도록 정의한다.
  • 각 업무별로 단독으로 사용되지 않고 모든 업무에 공통적으로 사용되는 코드에 대해서는 시스템명을 별도로 분리하여 작성하도록 한다. (예를 들어, 시스템명을 ‘공통’으로 분류하여 공통코드들을 정의)
Tailoring
Impact of not having

미작성시 영향

  • 데이터코드의 분류와 체계가 정의되지 않아 체계적인 데이터베이스 설계가 어렵다.
Reasons for not needing

작성이 불필요한 경우

  • 데이터코드목록은 반드시 작성하도록 한다.
Representation Options

제출시 고려사항

  • 설계단계 완료 시 고객에게 제출하고 승인을 받도록 한다.
