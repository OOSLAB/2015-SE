Main Description

작성항목

  1. 현행 시스템 개요 : 현행 시스템에 대한 상위수준의 구성 시스템을 도출하고, 각 시스템에 대한 업무적인 특징이나 아키텍처 관점의 특이사항 등을 기술한다.
  2. 현행 어플리케이션 아키텍처 현황 : 현재 사용중인 어플리케이션의 구성현황을 파악하기 위해 현행 어플리케이션 구성도, 어플리케이션 구조도, 상용 응용소프트웨어 및 소프트웨어 아키텍처 스타일 등을 작성하고 각 구성의 구성요소에 대한 현황을 기술한다.
  3. 현행 기술인프라 아키텍처 현황 : 현재 사용중인 현행시스템의 H/W, S/W, N/W 구성도를 작성하고, 각 구성항목에 대한 사양(H/W 기종, OS, DBMS 등)을 기술한다.
  4. 현행 데이터 아키텍처 현황 : 현재 운영 중인 시스템의 데이터베이스 구조를 파악하기 위해 데이터베이스 구성도를 작성하고, 주제영역별로 테이블을 분류하여 테이블 명, 건수 등의 내용을 포함하여 기술한다.
  5. 현행시스템 연계 현황 : 현재 운영 중인 시스템의 내/외부 연계방식과 연계대상 현황을 작성하고, 연계대상 별로 연계방법 및 주기 등을 작성한다.
  6. 현행시스템 정보운영 현황 : 현행시스템에 대한 운용조직, 장애대응/백업/보안 정책, 표준화 정책, 시스템 사용현황 등의 정보운영현황을 작성한다.
  7. 현행시스템 문제점 : 현행시스템 분석결과 도출된 주요 문제점을 아키텍처 관점에서 기술한다.

Key Considerations

작성시 고려사항

  • 현행시스템을 설명하고 있는 문서(현행시스템의 분석/설계서)를 최대한 확보한다. 분석가가 분석에 필요한 사항 이상으로 과도하게 현행시스템을 분석할 필요는 없으며, 제안된 신 시스템의 속성의 범위 내에서 현행시스템분석서를 작성하도록 한다.
  • 기능목록, 테이블목록 등의 자료가 고객에서 입수 가능한 경우 별첨 형태로 관리하여도 무방하다. 관련 전자문서 외에 파일이 없는 경우 출력물 문서 그대로 활용 가능하다.
  • 현행시스템 문제점의 경우 현행시스템의 전반적인 주요 문제점을 기술하도록 하며, 현행시스템 분석결과 도출된 문제점은 아키텍처 설계에서 그에 대한 해결방안이 제시되어야 한다.
  • 목표 아키텍처 구성을 위한 구성요소에 해당하는 부분은 모두 빠짐없이 현황을 분석하여 아키텍처 설계의 실효성을 떨어뜨리지 않도록 한다.
Tailoring
Impact of not having

미작성시 영향

  • 현행시스템의 충분한 분석 없이 아키텍처 설계를 수행하는 경우 현행시스템의 이슈 등이 고려되지 않아 아키텍처 설계의 실효성이 떨어질 수 있으며, 아키텍처 설계단계에서 현행시스템의 요건들을 고려하기 위한 불필요한 반복/산발적인 작업이 수행될 수 있다.
Reasons for not needing

작성이 불필요한 경우

  • 현행시스템이 존재하지 않거나 ISP 등의 수행 산출물을 이용하여 현행시스템의 현황에 대해 충분한 파악이 가능한 경우
