Work Product (Artifact): 테이블정의서
물리적 데이터 모델링 단계에서 도출된 테이블들에서 관리되는 컬럼들의 특징(컬럼명, 데이터 타입, 길이, Null여부, Key여부, Default값), 인덱스, 업무규칙을 문서화한다.

Main Description

작성항목

  1. 주제영역명 : 논리 및 물리 데이터 모델링 시 분류한 주제영역. 테이블 목록의 주제영역명과 동일
  2. 테이블ID : DBMS에서 구현되는 물리적인 테이블 명칭. 테이블이 아닌 파일일 경우에는 파일ID 기재 (테이블 목록에 정의된 테이블ID)
  3. 테이블명 : DBMS에서 물리적으로 구현되는 테이블에 대한 한글명(일반적으로 논리적 데이터 모델의 엔티티명임). 테이블이 아닌 파일일 경우에는 파일명 기재 (테이블 목록에 정의된 테이블명)
  4. 테이블설명 : 테이블의 역할, 특징 등 테이블에 대한 설명
  5. No. : 컬럼의 순서를 일련번호로 기재
  6. 컬럼ID : DBMS에서 물리적으로 구현되는 컬럼 명칭
  7. 컬럼명 : DBMS에서 물리적으로 구현되는 컬럼의 한글명
  8. 타입 : 컬럼이 갖는 데이터 타입. 해당 데이터베이스에서 지원하는 데이터 타입을 기준으로 기재
  9. 길이 : 컬럼의 길이를 바이트수로 기재
  10. NULL : 컬럼의 NULL 허용여부 (Not Null일 경우 ‘N’, Null을 허용할 경우 ‘’)
  11. KEY : 컬럼이 KEY인 경우 기재 (PK, FK)
  12. DEFAULT : 컬럼이 기본 값을 갖는 경우 DEFAULT값을 기재
  13. 비고 : 해당 컬럼에만 해당하는 특징이 있는 경우 기재
  14. 인덱스 : 테이블에 생성되는 인덱스
  15. 인덱스키 : 인덱스를 구성하는 컬럼들
  16. 업무규칙 : 테이블에 해당하는 업무규칙을 기재 (입력규칙, 수정규칙, 삭제규칙)

Key Considerations

작성시 고려사항

  • 테이블ID, 컬럼ID의 길이가 너무 길 경우 테이블 생성 시 오류가 생길 수 있으므로, 데이터베이스가 허용하는 길이를 넘지 않도록 한다.
  • 주키(PK) 이외의 컬럼들을 모두 Null허용으로 정의하는 경우가 많은데, 이는 컬럼의 특징과 달리 잘못 정의된 경우가 많다. 주키(PK) 이외의 컬럼들에 대해 각각의 특징을 정확히 파악하여 Null여부를 정의하도록 한다.
  • 컬럼의 데이터 길이가 컬럼 데이터의 최대 길이보다 작게 정의했을 경우, 테이블 생성 후 데이터 입력 시 오류가 생길 수 있으므로 컬럼 데이터의 최대 길이를 만족하는 길이로 정의하도록 한다.
  • 테이블목록에 정의한 모든 테이블에 대해 각 테이블별로 테이블정의서를 작성한다.
  • 각 컬럼의 데이터 타입은 해당 데이터베이스가 지원하는 데이터 타입을 기준으로 작성하도록 한다.
  • 컬럼이 DEFAULT값을 갖는 경우, DEFAULT값을 정확히 명시하도록 한다.
  • 컬럼ID는 자료사전에 정의된 표준용어를 기준으로 정의하도록 한다.
  • 테이블 설명에 간단한 내용이라도 작성하여 테이블의 역할에 대해 한눈에 알 수 있도록 한다.
Tailoring
Impact of not having

미작성시 영향

  • 물리 DB의 생성 및 관리가 어렵다.
  • DBMS의 특성 및 성능/보안 등을 고려한 튜닝정보(예:인덱스)를 파악하기 어렵다.
  • 정확한 테이블구조, 컬럼 속성, 업무규칙 등을 파악하기 어려워 프로그램 개발시 일관성 및 코딩의 효율성이 떨어지게 된다.
Reasons for not needing

작성이 불필요한 경우

  • 테이블목록은 반드시 작성하도록 한다.
Representation Options

제출시 고려사항

  • 설계단계 완료 시 고객에게 제출하고 승인을 받도록 한다.
