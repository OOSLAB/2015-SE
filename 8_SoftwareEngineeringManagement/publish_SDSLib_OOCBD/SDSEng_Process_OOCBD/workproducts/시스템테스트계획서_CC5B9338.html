Description
Main Description

작성항목

  1. 개요
    • 시스템 테스트 목적 : 테스트를 수행하는 목적을 간략히 기술한다.
    • 테스트 영역
      • 어느 분야에 대한 시스템 테스트를 수행할지에 대해 기술한다. 시스템 테스트 중 기능적 시스템 테스트에 대한 상세 계획과 결과는 통합테스트 계획서 및 결과 보고서 내용과 동일하게 작성한다. 비기능적 시스템 테스트 중 여기서는 성능테스트에 대해서만 언급하고 이외의 비기능적 테스트는 Vendor 측에서 제공하는 계획서 및 결과서를 첨부할 수 있다.
      • 각 테스트 영역별로 요구사항 정의서에 기술되어 있는 관련된 요구Id와 연결 시킨다.
    • 테스트 전제조건 : 시스템 테스트 수행 전 미리 준비되어야 할 조건에 대하여 기술한다. 독립된 환경, 완성된 테스트 대상 시스템, 보안문제, 테스트용 데이터, 테스트용 장비 등에 관한 사항이 있을 수 있다.
    • 테스트 산출물 : 시스템 테스트 시 작성되어야 할 산출물을 기재한다. 시스템 테스트 계획서와 시스템 테스트 결과 보고서가 이에 해당하며 별도로 일일 계획서, 일일 결과서 등이 포함 될 수 있다.
    • 책임과 역할 : 시스템 테스트수행에 필요한 조직 및 구성원, 역할에 대하여 기술한다. 시스템 테스트와 관련된 프로젝트 팀원뿐 아니라 외부 지원조직, H/W, S/W 벤더 등도 다 포함한다.
    • 테스트 일정 : 각 태스크 별 상세 일정에 대하여 기술한다.
  2. 테스트 수행 방법
    • 테스트 수행 툴 : 시스템 테스트에 사용되는 Tool 종류 및 그에 대한 간략한 설명을 기술한다.
    • 측정 방식 : 시스템 테스트 시 사용되는 용어 및 측정 방법에 대하여 기술한다. 중요한 요소 중의 하나는 용어에 대한 명확한 정의가 필요하다는 것이다. 같은 용어를 놓고 고객과 프로젝트 팀 간의 이견이 발생할 수도 있기 때문이다.
  3. 테스트 접근 방법 및 최적화 방안
    • 대상 업무 선정 기준
      • 기능적 시스템 테스트는 시스템 전체에 걸친 통합테스트를 할 수 있어야 한다.
      • 비기능적 시스템 테스트 시 가장 크게 고려해야 할 사항은 사용 빈도가 높거나 피크가 발생할 수 있는 업무, 대용량 데이터를 발생시키거나 문제 발생시 치명적인 업무 등은 반드시 테스트 대상에 포함시켜야 한다.
    • 테스트 수행절차
      • 기능적 시스템 테스트는 개발 단계에서 점진적으로 수행하던 통합테스트를 전체 시스템을 통합한 후에 수행하는 개념으로만 바꾸어서 기술하면 된다.
      • 비기능적 시스템 테스트를 수행하기 위한 일반적인 절차는 시스템 및 요구사항 분석 – 시나리오 작성 – 테스트 실행 및 결과 분석 – 문제점 보완 – 재테스트 – 결과정리 의 순으로 진행하며 재테스트 시 추가적으로 문제점이 발생하면, 일정 수준을 만족할 때 까지 반복하여 테스트 한다.
    • 결함관리 방안 : 기존 테스트 계획서 상에 기술된 결함관리 방안과 동일하다. 그대로 활용해도 무관하다.
    • 성능 최적화 방안 : 비기능적 시스템 테스트 시 성능을 최적화 시키기 위한 방안에 대하여 상세히 기술한다. 테스트 실행 시 Web/WAS/DB/NW 등 다른 부분도 모니터링 하여 문제점을 도출한다.
    • 모니터링 계획 : 비기능적 시스템 테스트 시 모니터링 대상에 대한 모니터링 방법 및 담당자에 대하여 기술한다.
    • 모니터링 항목 : 비기능적 시스템 테스트 시 모니터링 할 세부항목에 대하여 기술한다.
  4. 테스트 환경
    • 테스트 구성도 : 비기능적 시스템테스트가 수행되는 환경 구성을 가능하면 이미지로 표현한다.
    • H/W, S/W 구성도 : 비기능적 시스템 테스트가 수행되는 각 구성요소들의 H/W, S/W 사양에 대하여 상세히 기술한다. 테스트 환경은 가급적 실 환경과 동일하게 가져가고 부득이한 경우 실 환경의 축소판 형태로 테스트 환경을 구축하고 실 환경과의 비교를 통해 결과값을 환산한다.
    • 테스트 Tool 구성도 : 비기능적 시스템 테스트에 사용되는 Tool 및 사양에 대하여 상세히 기술한다. 특히 Player Agent 가 부족하지 않도록 장비 준비에 신경 쓴다.
  5. 테스트 시나리오 구성
    • 통합테스트 시나리오 : 기능적 테스트 시나리오는 기존에 점진적으로 완성되어지던 통합테스트 시나리오와 동일하며 재사용 가능하다.
    • 비기능적 테스트 대상 화면/ 테스트 시나리오
      • 앞에 기술된 대상업무 선정 기준에 의거하여 테스트 대상화면을 선정하고 이를 이용하여 실제 테스트를 수행할 시나리오를 작성한다. 이때 반드시 시스템 성능 목표치를 기술한다. 이때 성능 목표치는 단순히 3초 이내 라고 할 것이 아니고 평균 3초, 90%값 3초 등 정확한 표현을 사용한다. Active User, Concurrent User 등 사용하는 용어의 정의를 명확하게 하고, 부하량이 Active User 인지, concurrent User인지 등을 명확히 하여 추후 문제의 소지를 제거한다.
      • 각 시나리오에 대해 관련 요구 ID를 명시한다.
    • 비기능적 테스트 부하량 산정방법 : 시스템 테스트 시 사용되는 부하량 산정 기준에 대하여 기술한다. 부하량은 기존 시스템이 존재하는 경우 로그 분석을 통하여 산정하고 없는 경우 유사 사이트를 분석하여 산정한다.

Impact of not having

미작성시 영향

  • 시스템 테스트 전략이 수립되지 않아 프로젝트 특성 및 규모에 적합한 테스트 수행이 어렵다.
  • 원활한 시스템 테스트 수행을 위한 테스트 환경구축이 어렵다.
  • 시스템 테스트 수행을 위한 리소스(일정, 인력, 장비 등)의 조기확보가 어렵다.
  • 시스템 테스트 범위, 수행주체 등에 대한 고객의 합의를 득하지 못하여 테스트 수행 시 고객의 적극적인 참여를 유도하기 어렵다.
