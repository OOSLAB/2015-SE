Work Product (Artifact): 단위테스트정의서
단위테스트는 개발된 프로그램의 결함을 발견하기 위하여 수행하는 테스트이다. 개발한 내용에 대해 점검을 위하여 단위테스트정의서를 작성한다.

Main Description

작성항목

  1. 클래스명 : 단위테스트의 대상이 되는 클래스명을 입력한다. 단, 단위테스트 대상이 배치처리인 경우에는 배치프로그램명을 입력한다.
  2. 오퍼레이션명 : 단위테스트의 대상으로 식별된 클래스의 오퍼레이션중 실제 단위테스트의 대상인 테스트케이스로 식별된 오퍼레이션명을 입력한다. 단, 단위테스트 대상이 배치처리인 경우에는 배치프로그램의 함수명을 입력한다.
  3. 테스트 절차 : 성공하는 단위테스트뿐만 아니라 비즈니스 예외상황이 발생하는 경우를 포함한 테스트케이스를 작성한다.
  4. 테스트데이터 : 단위테스트 수행을 위한 테스트 데이터를 작성한다.
  5. 예상결과 : 단위테스트 수행 후 예상되는 결과를 작성한다.

작성스타일

  • 일반적으로 텍스트 형태로 작성을 하지만, 만약 개발자가 중요한 단위테스트가 어떤 경우라는 것을 잘 알고 있다면 문서형태의 명세서를 작성하는 대신에 코드에 주석으로 추가할 수 있다.

Key Considerations

작성시 고려사항

  • 테스트케이스 작성시 Positive Case나 일반적인 Case만을 작성하지 않도록 한다.
  • 에러를 발생시킬 수 있은 상황이나 Negative Case를 테스트 할 수 있도록 한다.
  • 더 많은 테스트 케이스 시나리오를 작성할수록 소프트웨어의 품질은 더 좋아진다.
  • 단순히 테스트 커버리지를 높이기 위해서 일반적이고 간단한 테스트 케이스를 양산하지 않도록 한다.
  • 단위테스트 정의서는 문서형태가 아니더라도 소스코드 주석이나 모델의 설명으로 작성할 수 있다.
  • 테스트 범위는 단위기능에 국한되어야 한다.
  • 테스트 절차가 너무 복잡하여 테스트에 대한 테스트가 필요하면 안된다.
  • 테스트를 수행할 때 특별한 초기상태를 요구하는 경우 테스트 절차에 이를 명시하여야 한다.
  • 단위테스트 대상에 대한 식별기준은 프로젝트 별로 다를수 있기 때문에 이에 대한 기준이 필요하다. 따로 문서를 작성하지 말고 테스트 계획서에 추가하도록 한다.
  • 정상적인 테스트케이스를 도출할때는 블랙박스 테스트기법, 화이트 박스 테스트 기법, 극단적인 조건기법과 같은 테스트 기법을 참고하여 도출한다.
  • 데이터를 생성하는 경우뿐만 아니라 수정,삭제하는 경우에 대한 테스트 케이스가 도출되어야 한다.
  • 조회의 경우에도 데이터가 있을 때 목록이 조회되는 경우와 데이터가 얿을 때 데이터가 없음을 표시하는 경우도 테스트케이스로 도출되어야 한다.
  • 배치기능에 대한 단위테스트를 수행하기 위해서 이 단위테스트정의서를 활용할 수 있다.

Impact of not having

미작성시 영향

  • 프로젝트 환경에 맞는 점검항목들을 제시하지 못하여 단위테스트 시 확인되어야 할 기능이 중복적으로 테스트되거나 테스트 되지 않을 수 있다.
  • 적절한 단위테스트를 수행하지 않고 통합테스트를 수행하게 되었을 때 예상치 못한 에러가 발생할 수 있다.
Reasons for not needing

작성이 불필요한 경우

  • 단위테스트를 수행주체는 개발자이다. 만약 개발자가 중요한 단위테스트가 어떤 경우라는것을 잘 알고 있다면 문서형태의 명세서를 작성하는 대신에 코드에 주석으로 추가할 수 있다.
