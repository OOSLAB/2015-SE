Main Description

주요 작성항목

  1. 인터페이스 ID : 시스템인터페이스 목록에 있는 인터페이스 ID를 기재한다.
  2. Source 담당자 : 인터페이스 구축시 데이터를 Target으로부터 가져와서 사용할 Source 시스템의 인터페이스 담당자를 기재한다.
  3. Target 담당자 : 인터페이스시 필요한 데이터가 있는 Target 시스템에서의 인터페이스 담당자를 기재한다.
  4. 기능 요건 : 인터페이스를 구축할 때 구현 해야 할 기능 요건에 대해 기술한다.
  5. 인터페이스 유형정보
    • Source 시스템 : 인터페이스를 통해 데이터를 가져와서 이용할 시스템명을 기재한다.
    • Target 시스템 :인터페이스를 통해 가져올 데이터가 있는 시스템명을 기재한다.
    • 인터페이스 구분: 인터페이스가 내부시스템간 혹은 내/외부 시스템간에 발생하는지를 파악하여 대내/대외 여부를 기재한다.
    • 인터페이스 방식 : 웹서비스, FTP, DB Link, Socket 등 아키텍처에서 정의한 타 시스템과의 인터페이스 방식을 기재한다.
    • 통신 유형 : Request-Reply, Send-Receive, Send-Receive-Acknowledge, Publish-Subscribe 등 아키텍처에서 정의한 통신 유형을 참조하여 기재한다.
    • 처리 유형 : 실시간, 배치, 지연처리 등 인터페이스 처리 유형에 대해 기재한다.
    • 주기 : 인터페이스가 발생하는 주기를 기재한다.
    • 데이터포맷 : 고정길이, XML 등 인터페이스 항목의 데이터 포맷에 대해 기재한다.
    • 접근 경로 : 인터페이스 방식을 고려하여 Target시스템에 접근하기 위한 접근 정보를 기재한다.
    • 비고: 기타 도움이 될만한 사항을 기재한다.
  6. 인터페이스 데이터 항목
    • 필드명 : 한글 및 영문으로 된 인터페이스 데이터 항목을 기재한다.
    • 데이터 타입 : Character, Number 등의 데이터 타입을 기재한다.
    • 길이 : 해당 필드의 길이 정보를 기재한다.
    • 설명 : 해당 필드에 대한 설명을 기술한다. 특히, 해당 필드값이 코드화되어 있거나 필드값에 대한 처리 로직이 있는 경우 기술한다.
    • 비고 : 기타 도움이 될만한 내용을 기재한다.

Key Considerations

작성시 고려사항

  • 시스템인터페이스정의서 산출물은 인터페이스방식을 고려하여 테일러링되어야 한다.
  • 시스템인터페이스의 기능 특성을 고려하여 기능요건을 기술한다. 특히 처리 성격의 시스템인 경우, Target시스템에서의 처리로직이 기능요건에 포함되어야 한다.
  • 인터페이스 유형 정보는 시스템인터페이스목록과 일관되어야 한다.
  • 접근 경로는 인터페이스 방식을 고려하여 정의한다. 예를 들어, 웹서비스인 경우에는 URL정보를, FTP방식인 경우에는 Target 시스템 IP, 사용자계정/비밀번호, 디렉토리/파일정보 등을 정의한다.
  • 인터페이스 데이터 항목은 주고 받는 데이터 필드에 대한 정보를 정의한다. 입출력 데이터 항목이 유사한 경우에는 하나로 통합하여 작성할 수 있으나, 통합하여 작성할 수 없는 경우에는 인터페이스정의서를 요청과 응답에 대해 각각 작성하고 하나의 Set임을 명시한다.
  • 인터페이스 데이터 항목에는 모든 시스템인터페이스에 적용되는 공통데이터 항목은 포함시키지 않아도 된다.
  • 데이터 항목에서 반복되는 필드는 반복여부를 표시한다. 데이터 포맷이 XML형태인 경우에는 반복필드를 포함하고 있는 엘리먼트명을 명시한다.
  • 필드에 대한 설명 항목에는 해당 필드가 코드형태이거나, Target시스템에서 해당 필드에 대한 처리로직을 갖는 경우 기술한다. 특히, 대외기관과의 연계 시에는 각 코드의 의미를 기술해야 한다.
  • 인터페이스 유형 정보는 아키텍처 및 인터페이스 표준을 준수하여 정의한다.
  • 연계솔루션에서 Source시스템에서 전송한 데이터를 변경하거나 첨삭하여 Target시스템에 전송하는 경우, Source시스템과 연계솔루션간의 데이터항목과 연계솔루션과 Target시스템간의 데이터항목은 구분되어야 한다.

Impact of not having

미작성시 영향

  • 시스템 내부 및 타 시스템과의 인터페이스에 의해 전달되는 정보들이 손실되거나 변조될 위험이 있다.
  • 인터페이스가 명확히 정의되지 않으므로 개발 및 통합테스트 시 해당 기능이 누락될 수 있다.
  • 인터페이스 개발에 필요한 정보가 정의되지 않아 인터페이스 개발에 어려움이 있다.
  • 대외 기관과의 인터페이스 표준이 준수되지 않아, 해당 기관과의 인터페이스가 원활하지 않을 수 있다.
Reasons for not needing

작성이 불필요한 경우

  • 개발해야 할 내/외부 인터페이스가 없는 경우, 표준 인터페이스를 사용해야 하는 경우 등을 제외하고는 반드시 작성하도록 한다.
