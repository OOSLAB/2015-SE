Description
Main Description

작성항목

  1. 화면레이아웃 종류 식별 : 화면 프로토타입을 참조하거나 업무 내용상으로 어떤 종류의 화면이 필요한지를 식별한다. 예를 들어 목록 조회 화면, 탭이 있는 화면, 상세 조회 화면이면서 정보가 그룹별로 묶여 있는 화면 등 다양한 형태의 화면 종류를 들 수 있다.
  2. 화면 네비게이션 스타일 정의
    • 업무프로세스 스타일에 따라 : 기본적인 CRUD (등록,조회,수정,삭제) 작업의 흐름에 따른 네비게이션 스타일, 그 외 업무 프로세스에 따라 달라지는 네비게이션 스타일을 정의한다. 예를 들어 일괄 변경 업무와 특정 대상을 선택하여 변경하는 업무는 화면 네비게이션이 다르게 정의된다.
    • 화면 스타일에 따라 : 어떤 작업 후에 화면이 변경되게끔 구성된 화면과, 변경되지 않고 해당 화면이 다시 로드 되도록 구성된 화면 등 화면 구성 스타일에 따라서도 네비게이션 스타일을 정의할 수 있다. 예를 들어 여러 개의 탭으로 구성된 화면에서 각 탭별로 개별 등록 작업을 수행할 수 있고, 개별 등록 작업이 수행된 후에는 다른 화면으로 이동하지 않고 등록 성공 메시지만 나오는 경우와, 여러 탭의 내용을 한꺼번에 등록하고 등록 후에는 등록 결과 조회 화면으로 이동하는 경우를 들 수 있다.
  4. 화면 디자인 제약 사항 식별 : 고객사의 CI 정책에 따른 색상 프로파일/폰트 등이 제한, 디스플레이 기기의 제약으로 인한 해상도 제한, 네트워크 속도 문제로 인한 이미지 파일 크기 제한 등 각 고객에 특화된 사유로 인한 화면 디자인에서의 제한 요소들을 UI 기획자에게 알려주어야 한다.
  5. 이벤트 표준 정의 : 공통적으로 사용하는 화면이나 화면 구성요소에서 발생하는 이벤트명과 입/출력값 등을 표준화하여 정의한다. 예를 들어 ‘고객번호조회’ 이벤트의 경우 입력값은 주민번호, 고객ID이고 출력값은 고객번호로 정의한다.

Key Considerations

작성시 고려사항

  • 구축할 시스템의 전체 화면에 적용할 표준이다. 화면 구성이 다른 서브 시스템이 있는 경우, 각각의 서브 시스템 별로 정의한다.
  • 화면 구성에서 통일감 / 심미성을 위한 시각 디자인적 요소는 UI 기획자가 표준을 정하는 것이 적합하다.
  • 고객이 직접 사용하는 화면에 전반적으로 영향을 미치는 표준이므로, 화면 표준은 고객 승인을 받아야 한다.
Tailoring
Impact of not having

미작성시 영향

  • 개발 시 화면에 대한 표준이 없이 작업되어 전체적인 화면의 일관성을 유지하기 어렵다.
  • 고객과 화면에 대한 합의가 이루어지지 않아 개발 후기 잦은 고객 변경요청의 원인이 된다.
